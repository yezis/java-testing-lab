# 学习方式与改进约定

这份文档记录本项目后续学习时采用的方式。目标是让学习不只是“跟着写代码”，而是逐步形成独立理解、独立排查、独立设计测试的能力。

## 1. 每节结束做小复盘

每学完一个测试知识点，都固定回答三个问题：

```text
1. 这个工具或写法解决什么问题？
2. 当前项目里哪一行代码用了它？
3. 如果不用它，测试会变成什么样？
```

例如学习 `ArgumentCaptor` 时，要能说明：

```text
它用于捕获传给 Mock 方法的复杂对象，从而验证对象内部字段是否正确。
```

对应当前项目：

```text
OrderServiceTest 中用它捕获传给 orderRepository.save(...) 的 Order 对象。
```

## 2. 测试代码保持干净

学习过程中可以临时保留旧写法用于对比，但最终测试文件里只保留当前推荐写法。

不建议长期保留这种注释代码：

```java
// when(...).thenReturn(...);
```

原因：

- 测试代码会越来越难读
- 容易分不清哪段是当前有效写法
- 真实项目中测试代码也需要维护质量

旧写法、对比写法、解释说明应该放在学习文档里，而不是长期留在测试类里。

## 3. 新知识点优先新增代码，不反复修改旧代码

每次学习一个新的知识点时，优先采用下面方式：

```text
1. 新增一个测试方法
2. 新增一个专门用于演示的小类
3. 新增一个独立的示例测试类
```

尽量不要反复修改已经写好、已经通过的测试代码。

这样做的好处：

- 之前学过的内容可以保留下来，方便回顾
- 每个知识点都有独立示例
- 测试文件能体现学习路径
- 不会因为学习新 API 而覆盖旧 API 的示例

例如学习 `Spy` 时，不优先把已有的 `OrderServiceTest` 大幅改掉，而可以新增一个测试类：

```text
OrderCalculatorSpyTest
```

或者新增一个专门的测试方法来演示 `Spy` 的行为。

只有在下面情况才修改旧代码：

- 旧代码本身有错误
- 旧代码影响后续测试运行
- 用户明确要求重构或替换
- 为了保持项目可维护性，需要做小范围清理

## 4. 测试失败时先分析，不急着改

每次测试失败，先按下面顺序判断：

```text
1. 是编译失败，还是测试运行失败？
2. 如果是断言失败，期望值是什么？
3. 实际值是什么？
4. 期望值和实际值差在哪里？
5. 这个差异来自业务代码、测试代码，还是测试数据？
```

例如保存失败测试曾经出现过：

```text
期望异常：IllegalStateException
实际异常：IllegalArgumentException
```

这说明不是 Mockito 用法错，而是 Stub 中设置的异常类型和断言期望不一致。

## 5. 逐步减少照抄，增加测试设计

前期可以直接给完整代码，帮助建立基本手感。

后续应逐步改成：

```text
1. 先给测试目标
2. 给关键 API
3. 学习者先写测试代码
4. 再 review 代码和测试结果
```

例如：

```text
目标：保存订单失败时，Service 应该把异常继续抛出。
关键 API：doThrow、assertThatThrownBy、verify(..., times(1))
```

这种方式更接近真实工作，也能训练测试设计能力。

## 6. 每个阶段要及时收束

不要在一个工具的高级特性里停留太久。

Mockito 的学习重点是：

- 会隔离依赖
- 会准备依赖返回值
- 会验证依赖调用
- 会测试异常分支
- 能理解 Mock、Stub、Spy 的区别
- 能看懂常见失败提示

学完这些后，应进入 Spring Boot 测试和集成测试，而不是长期停留在 Mockito 高级 API。

## 7. Fuzzing 放在基础测试之后

Fuzzing 已经加入学习体系，但不应太早展开复杂案例。

推荐顺序：

```text
JUnit 5 -> Mockito -> Spring Boot 测试 -> 集成测试 -> 属性测试与 Fuzzing
```

Fuzzing 的第一个练习应优先选择纯函数或输入校验逻辑，例如：

```text
OrderCalculator
```

适合验证的属性：

- 合法价格和数量计算出的总价不应为负数
- 数量为 0 时总价应为 0
- 非法输入应该稳定抛出异常
- 同样输入应该得到同样输出

## 8. 维护测试知识索引

后续可以新增：

```text
docs/testing-cheatsheet.md
```

用于记录常用测试写法，例如：

```text
断言普通结果：assertThat(...)
断言异常：assertThatThrownBy(...)
Mock 返回值：when(...).thenReturn(...)
Mock 抛异常：when(...).thenThrow(...)
验证调用：verify(...)
验证不调用：never()
捕获参数：ArgumentCaptor
验证顺序：InOrder
```

长文档用于理解，索引用于快速查写法。

## 9. 后续学习时的默认流程

每个知识点按这个节奏推进：

```text
1. 先解释概念和适用场景
2. 在当前项目中找到合适业务位置
3. 给出测试目标，而不是直接堆代码
4. 学习者实现测试
5. 运行 mvn test
6. 根据结果分析失败或确认通过
7. 把新知识点写入文档
8. 简短复盘
```

这个流程可以兼顾理解、实践、排错和沉淀。
